## Шаг 3. Модели данных и связи между сущностями

Нужно заранее зафиксировать все основные сущности приложения и их связи, чтобы дальше не переделывать структуру данных во время разработки.  
Модели должны быть простыми, понятными и достаточными для MVP.

### Основные сущности

В приложении используются 4 основные сущности:

- Project
- Task
- WorklogEntry
- TaskComment

---

## 1. Project

Project — это проект, внутри которого живут задачи.

### Поля Project

- `id: string`
- `name: string`
- `code: string`
- `description?: string`
- `color: string`
- `createdAt: string`

### Правила для Project

- `id` должен быть уникальным
- `code` задаётся вручную и используется как короткая аббревиатура проекта
- `code` должен быть коротким и удобным для поиска
- пример: `PRO1`, `ERP`, `OPS`
- `color` используется для визуального выделения проекта в интерфейсе
- `createdAt` хранится в виде строки даты

---

## 2. Task

Task — это задача, которая принадлежит одному проекту.

### Поля Task

- `id: string`
- `projectId: string`
- `code: string`
- `title: string`
- `description?: string`
- `status: 'planning' | 'progress' | 'done'`
- `estimatedHours?: number`
- `createdAt: string`

### Правила для Task

- каждая задача обязательно принадлежит одному проекту
- связь задаётся через `projectId`
- `code` должен включать код проекта и номер задачи
- пример: `PRO1-001`, `ERP-003`
- `status` может быть только одним из трёх значений:
  - `planning`
  - `progress`
  - `done`
- `estimatedHours` не обязателен
- завершённые задачи не должны появляться в выборе при создании новых worklog-блоков на недельной доске
- завершённые задачи не удаляются из данных и должны участвовать в аналитике

---

## 3. WorklogEntry

WorklogEntry — это запись о работе по задаче в конкретный день и конкретное время.

### Поля WorklogEntry

- `id: string`
- `taskId: string`
- `date: string`
- `dayIndex: number`
- `startHour: number`
- `durationHours: number`
- `comment?: string`

### Вычисляемое поле

Поле `endHour` не нужно хранить отдельно в данных.  
Оно должно вычисляться в коде по формуле:

`endHour = startHour + durationHours`

### Правила для WorklogEntry

- каждая запись обязательно принадлежит одной задаче
- связь задаётся через `taskId`
- `date` хранит конкретную календарную дату
- `dayIndex` нужен для удобной привязки к колонке недели
- использовать следующую нумерацию:
  - 1 — понедельник
  - 2 — вторник
  - 3 — среда
  - 4 — четверг
  - 5 — пятница
  - 6 — суббота
  - 7 — воскресенье
- `startHour` задаётся вручную
- `durationHours` задаётся вручную
- `comment` у worklog-записи необязателен
- одна и та же задача может иметь несколько worklog-записей в течение одной недели
- worklog-записи используются для расчёта потраченного времени по задачам и проектам

---

## 4. TaskComment

TaskComment — это отдельный комментарий на странице задачи.

### Поля TaskComment

- `id: string`
- `taskId: string`
- `text: string`
- `createdAt: string`

### Правила для TaskComment

- каждый комментарий принадлежит одной задаче
- связь задаётся через `taskId`
- комментарии хранятся отдельно от worklog-записей
- комментарии нужны только для страницы задачи
- редактирование и удаление комментариев в MVP не требуется

---

## Связи между сущностями

### Project → Task
Один проект может содержать много задач.

Связь:
- `Task.projectId -> Project.id`

### Task → WorklogEntry
Одна задача может содержать много записей worklog.

Связь:
- `WorklogEntry.taskId -> Task.id`

### Task → TaskComment
Одна задача может содержать много комментариев.

Связь:
- `TaskComment.taskId -> Task.id`

---

## Что считается вычисляемым, а не хранимым

Следующие значения не нужно сохранять отдельными полями в данных.  
Они должны вычисляться через сервисы или утилиты:

- `endHour`
- общее количество часов по задаче
- общее количество часов по проекту
- количество задач в проекте
- часы по неделе
- часы по дням недели
- часы по задачам внутри проекта

---

## Ограничения MVP по моделям

- не добавлять subtasks
- не добавлять пользователей
- не добавлять роли
- не добавлять месячную аналитику как отдельную сущность
- не хранить сложные статусы, кроме:
  - `planning`
  - `progress`
  - `done`
- не добавлять отдельную сущность для календаря
- не хранить `endHour` в базе данных отдельно
