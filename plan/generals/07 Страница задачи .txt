## Шаг 7. Страница задачи

Нужно реализовать отдельную страницу задачи, на которой пользователь сможет посмотреть детали задачи, изменить её статус, добавить комментарий и увидеть, сколько времени уже было потрачено на эту задачу.

Маршрут страницы:

`/tasks/:id`

Эта страница должна быть простой, полезной и логичной.  
Она не должна превращаться в тяжёлую систему управления задачами.  
Задача этого экрана — показать аккуратную детализацию одной сущности.

---

## Назначение страницы

Страница задачи нужна для:

- просмотра полной информации по задаче
- изменения статуса задачи
- добавления комментариев
- просмотра потраченного времени
- перехода обратно к проекту

---

## Верхняя часть страницы

В верхней части страницы нужно показать основную информацию о задаче.

### Что должно быть видно

- код задачи
- название задачи
- проект, к которому относится задача
- статус задачи
- описание, если есть
- estimated hours, если указано
- spent hours
- кнопка или ссылка возврата на страницу проекта

### Важно

`spent hours` должно вычисляться автоматически на основе всех worklog-записей, связанных с этой задачей.

---

## Статус задачи

На странице задачи должна быть возможность изменить статус.

### Допустимые статусы

- `planning`
- `progress`
- `done`

### Требования

- статус должен отображаться заметно
- статус должен быть окрашен визуально
- изменение статуса должно сохраняться в `localStorage`
- после изменения статус должен сразу обновляться в интерфейсе

### Цветовая индикация

Нужно использовать единый визуальный стиль статусов во всём приложении:

- `planning`
- `progress`
- `done`

Стиль должен совпадать с тем, что используется в каталоге и на странице проекта.

---

## Комментарии к задаче

На странице задачи должен быть отдельный блок комментариев.

### Что нужно реализовать

- список комментариев
- форма добавления нового комментария

### Поля нового комментария

- `text`

### Поведение

- комментарий привязывается к текущей задаче
- после добавления комментарий сразу появляется в списке
- комментарий сохраняется в `localStorage`
- комментарии отображаются в порядке добавления, от новых к старым или от старых к новым — выбрать один вариант и использовать его везде последовательно

### Что не нужно

- не нужно редактирование комментариев
- не нужно удаление комментариев
- не нужны вложенные комментарии

---

## Потраченное время по задаче

На странице нужно явно показать суммарное время по задаче.

### Как считается

Сумма всех `durationHours` у worklog-записей, связанных с этой задачей.

### Что нужно показать

- общее количество часов по задаче
- при желании — количество worklog-записей

### Но

Не нужно на этом шаге делать отдельную сложную таблицу всех worklog-записей задачи, если это перегружает MVP.

Главное — показать итоговое количество времени.

---

## Связь с проектом

На странице задачи нужно показывать, к какому проекту она относится.

### Что требуется

- показать название проекта
- показать код проекта
- дать возможность вернуться на страницу проекта

Это важно для навигации и для понимания контекста задачи.

---

## Загрузка страницы

При открытии `/tasks/:id` нужно:

- найти задачу по `id`
- найти проект этой задачи
- загрузить комментарии по `taskId`
- загрузить worklog-записи по `taskId`
- вычислить `spent hours`

---

## Поведение, если задача не найдена

Если задача по `id` не найдена, нужно показать корректное пустое или error-состояние.

### Что допустимо

- сообщение “Задача не найдена”
- кнопка возврата в каталог проектов

---

## Рекомендуемая структура страницы

Страница может состоять из следующих блоков:

- верхний блок с основными данными задачи
- блок статуса
- блок времени
- блок комментариев
- форма добавления комментария

---

## Что должно сохраняться

На этой странице должны сохраняться:

- изменение статуса задачи
- новые комментарии

Оба типа данных должны сразу записываться в `localStorage`.

---

## Что не нужно делать на этом шаге

- не добавлять редактирование всей задачи целиком
- не добавлять удаление задачи
- не добавлять subtasks
- не добавлять историю изменений
- не добавлять сложную ленту активности
- не превращать комментарии в чат
- не делать отдельный редактор описания

---

## Ожидаемый результат шага

После завершения этого шага должно быть реализовано:

- страница `/tasks/:id`
- отображение основных данных задачи
- показ проекта, к которому относится задача
- показ статуса задачи
- возможность изменить статус
- показ суммарного времени по задаче
- список комментариев
- форма добавления комментария
- сохранение статуса и комментариев в `localStorage`
