## Шаг 11. Финальная полировка MVP и критерии готовности

Нужно довести проект до состояния, в котором его не стыдно выложить на GitHub и показать на собеседовании как аккуратный, законченный и понятный mini pet-project.

На этом шаге не нужно добавлять новые крупные функции.  
Нужно собрать уже реализованные части в цельный, удобный и презентабельный MVP.

---

## Цель финального этапа

Финальный этап нужен для того, чтобы проект:

- выглядел завершённым
- был удобен для навигации
- имел понятные состояния интерфейса
- содержал реалистичные демо-данные
- был пригоден для показа работодателю
- не выглядел как набор несвязанных экранов

---

## Что должно быть приведено в порядок

На этом этапе нужно проверить и доработать:

- навигацию
- единый визуальный стиль
- статусы и цветовую индикацию
- пустые состояния
- seed-данные
- текстовые подписи интерфейса
- базовую UX-логику
- README и описание проекта

---

## 1. Навигация приложения

В приложении должна быть простая и понятная навигация между основными страницами.

### Основные маршруты

- `/projects`
- `/projects/:id`
- `/tasks/:id`
- `/board`
- `/analytics`

### Что требуется

- сделать верхнюю навигацию или боковое меню
- явно показать основные разделы приложения
- обеспечить быстрый переход между ключевыми страницами
- добавить удобные ссылки “назад”, где это нужно
- не допускать тупиковых экранов без понятного выхода

### Навигация должна быть понятной

Пользователь должен легко понимать:

- где находится каталог проектов
- где недельная доска
- где аналитика
- как вернуться из задачи в проект
- как вернуться из проекта в общий каталог

---

## 2. Единый визуальный стиль

Все экраны должны выглядеть как части одного приложения.

### Нужно привести к единому виду

- заголовки страниц
- кнопки
- карточки
- поля форм
- отступы
- блоки аналитики
- таблицы и списки
- пустые состояния

### Требования

- использовать единые размеры и отступы
- использовать единые стили кнопок
- использовать единые стили карточек
- не смешивать слишком много визуальных подходов
- не перегружать интерфейс декоративными эффектами

---

## 3. Статусы и цветовая индикация

Статусы задач должны отображаться одинаково во всём приложении.

### Статусы

- `planning`
- `progress`
- `done`

### Требования

- закрепить единый цвет для каждого статуса
- использовать одинаковый badge или визуальный индикатор
- показывать статус:
  - на странице проекта
  - на странице задачи
  - на недельной доске
  - в аналитике по задачам, если уместно

### Важно

Пользователь должен узнавать статус с первого взгляда, без чтения длинного текста.

---

## 4. Пустые состояния

Во всех ключевых местах должны быть корректные пустые состояния.

### Где они нужны

- каталог проектов
- список задач проекта
- неделя без worklog-записей
- день без записей на доске
- задача без комментариев
- проект или задача не найдены

### Пустое состояние должно быть

- коротким
- понятным
- спокойным
- без лишней графики

### Желательно

Если это уместно, добавить понятный CTA:
- “Создать проект”
- “Создать задачу”
- “Добавить запись”
- “Вернуться к проектам”

---

## 5. Seed-данные для демо

Seed-данные должны выглядеть реалистично и демонстрировать все основные возможности приложения.

### Что должны показывать seed-данные

- несколько проектов
- задачи в разных статусах
- worklog-записи на текущую неделю
- несколько комментариев к задачам
- заполненную аналитику
- заполненную недельную доску

### Требования к seed-данным

- данные должны быть понятными по названию
- не использовать случайный мусор
- коды проектов и задач должны выглядеть правдоподобно
- примеры задач должны быть близки к офисной или проектной работе

### Цель

Чтобы после первого запуска работодатель сразу видел:

- каталог проектов
- задачи
- weekly board
- аналитику
- детализацию задачи

без необходимости самому что-то заполнять руками.

---

## 6. Тексты интерфейса

Подписи в интерфейсе должны быть короткими, понятными и единообразными.

### Нужно проверить

- названия кнопок
- названия вкладок
- заголовки страниц
- подписи форм
- сообщения об ошибках
- пустые состояния

### Требования

- не использовать случайную смесь формулировок
- не использовать слишком технические подписи
- не использовать длинные абзацы текста в UI

---

## 7. Базовая UX-полировка

Нужно проверить, чтобы базовые сценарии были удобны и предсказуемы.

### Основные сценарии

- открыть каталог проектов
- найти проект
- открыть проект
- создать задачу
- открыть задачу
- изменить статус
- добавить комментарий
- открыть недельную доску
- добавить worklog-запись
- переместить запись на другой день
- открыть недельную аналитику

### Что нужно проверить

- после создания сущности она сразу видна в списке
- после изменения статуса UI сразу обновляется
- после добавления комментария он сразу появляется
- после drag and drop данные сохраняются
- после перезагрузки страницы все изменения остаются

---

## 8. README и оформление GitHub

Проект должен быть готов к публикации на GitHub.

### В README нужно указать

- название проекта
- короткое описание
- что это за приложение
- какой стек используется
- какие функции реализованы
- что данные хранятся через seed + localStorage
- как запустить проект локально

### Желательно добавить

- 2–4 скриншота интерфейса
- список основных страниц
- краткое пояснение, почему проект сделан именно так

### Важно

README должен быть коротким, чистым и понятным.  
Не нужно писать огромную документацию.

---

## 9. Что не нужно добавлять на финальном этапе

- не добавлять новые большие фичи
- не подключать backend в последний момент
- не добавлять chart-библиотеки
- не добавлять анимации ради анимаций
- не усложнять архитектуру
- не переписывать проект целиком

---

## 10. Критерии готовности MVP

Проект можно считать готовым, если выполнены следующие условия:

### Архитектура
- есть понятная структура папок
- данные разделены по сущностям
- storage-логика вынесена в сервисы
- страницы не перегружены бизнес-логикой

### Функциональность
- работает каталог проектов
- работает страница проекта
- работает создание задач
- работает страница задачи
- работает изменение статуса
- работают комментарии
- работает недельная доска
- работает создание worklog-записей
- работает drag and drop
- работает недельная аналитика

### Хранение данных
- seed-данные загружаются при первом запуске
- изменения сохраняются в `localStorage`
- после перезагрузки ничего не теряется

### UI
- статусы окрашены
- навигация понятна
- пустые состояния обработаны
- интерфейс выглядит аккуратно и единообразно

### Демонстрация
- проект можно показать без дополнительных объяснений
- после запуска интерфейс уже заполнен данными
- видно, что это не просто todo-list, а маленький рабочий инструмент

---

## Результат шага

После завершения этого шага проект должен быть:

- визуально цельным
- функционально завершённым в рамках MVP
- готовым к публикации на GitHub
- готовым к демонстрации на собеседовании
