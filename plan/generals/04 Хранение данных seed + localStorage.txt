## Шаг 4. Хранение данных: seed + localStorage

Нужно реализовать простую, надёжную и понятную схему хранения данных без backend и без настоящей базы данных.  
Для MVP все данные должны жить локально в браузере, но приложение должно запускаться уже с готовым демо-наполнением.

### Общий принцип

Использовать двухэтапный подход:

1. начальные демо-данные хранятся в seed-файлах
2. при первом запуске они копируются в `localStorage`
3. после этого приложение работает только с данными из `localStorage`

### Источник истины

После инициализации основным источником истины становится только `localStorage`.

Это означает:

- приложение не должно каждый раз заново читать seed-данные как рабочие
- seed нужны только для первого запуска
- все изменения пользователя должны сохраняться в `localStorage`

---

## Какие данные нужно хранить

В `localStorage` должны храниться отдельные коллекции:

- проекты
- задачи
- worklog-записи
- комментарии к задачам

### Рекомендуемые ключи localStorage

- `worklog-projects`
- `worklog-tasks`
- `worklog-entries`
- `worklog-comments`
- `worklog-seeded`

### Назначение ключа `worklog-seeded`

Это технический флаг, который показывает, была ли уже выполнена первичная инициализация seed-данных.

Пример:
- если `worklog-seeded` отсутствует → нужно выполнить инициализацию
- если `worklog-seeded = true` → инициализацию повторять не нужно

---

## Требования к seed-данным

Seed-данные должны быть реалистичными и демонстрационными.

Они должны включать:

- несколько проектов
- несколько задач внутри проектов
- несколько worklog-записей на неделю
- несколько комментариев к задачам

### Зачем нужны seed-данные

- чтобы приложение не выглядело пустым
- чтобы сразу показать структуру данных
- чтобы можно было сразу открыть доску, аналитику и страницы проекта
- чтобы GitHub-демо выглядело убедительно

### Пример минимального наполнения

- 3 проекта
- 8–12 задач
- 10–20 worklog-записей
- 3–6 комментариев

---

## Требования к инициализации

Нужно реализовать логику первого запуска.

### Поведение при старте приложения

#### Если localStorage пустой
Нужно:

- загрузить seed-данные
- записать их в `localStorage`
- поставить флаг `worklog-seeded = true`

#### Если localStorage уже содержит данные
Нужно:

- не перезаписывать существующие данные
- использовать то, что уже лежит в `localStorage`

### Важное правило

Приложение не должно уничтожать данные пользователя при повторном открытии страницы.

---

## Архитектура хранения

Логику хранения нельзя размазывать по компонентам и страницам.

Нужно выделить отдельные сервисы.

### Минимальный набор сервисов

#### `storage.service.ts`
Отвечает за базовую работу с `localStorage`:

- чтение по ключу
- запись по ключу
- удаление по ключу
- проверка наличия данных
- безопасный parse/stringify

#### `seed.service.ts`
Отвечает за первичную инициализацию:

- проверяет, выполнен ли seed
- если нет — загружает seed-данные
- записывает их через `storage.service`

---

## Требования к storage.service

Сервис должен:

- быть универсальным
- работать с generic-типами
- централизовать доступ к `localStorage`
- не содержать бизнес-логики проектов, задач и worklog
- обрабатывать отсутствие данных безопасно

### Что должен уметь storage.service

- `getItem`
- `setItem`
- `removeItem`
- `hasItem`
- `clearAllAppData` — опционально, для режима разработки
- безопасно возвращать `[]` или `null`, если данных нет

---

## Требования к seed.service

Сервис должен:

- запускаться один раз при старте приложения
- проверять наличие флага и данных
- инициализировать хранилище seed-данными
- не выполнять повторную инициализацию без необходимости

### Поведение seed.service

- если приложение запускается впервые — записать seed
- если данные уже есть — ничего не делать
- логика должна быть простой и прозрачной

---

## Формат работы с данными

Все feature-сервисы должны работать не напрямую с `localStorage`, а через `storage.service`.

Примеры:

- `projects.service` читает и обновляет проекты через `storage.service`
- `tasks.service` читает и обновляет задачи через `storage.service`
- `worklog.service` читает и обновляет worklog через `storage.service`

### Запрещено

- читать `localStorage` прямо в компонентах
- писать в `localStorage` прямо в страницах
- дублировать ключи в разных местах приложения
- смешивать seed-логику и feature-логику в одном сервисе

---

## Поведение при изменении данных

Все операции должны сразу сохраняться в `localStorage`.

Это касается:

- создания проекта
- создания задачи
- изменения статуса задачи
- добавления комментария
- создания worklog-записи
- перемещения блока на доске

### Ожидаемый результат

После перезагрузки страницы пользователь должен видеть все свои изменения.

---

## Отладочный режим

Для удобства разработки можно предусмотреть служебную функцию сброса данных приложения.

### Допустимо
Сделать временную dev-функцию:

- очистить все ключи приложения из `localStorage`
- заново инициализировать seed

### Но
Эта функция не должна быть частью основного UI для демонстрации, если она не нужна.

---

## Результат шага

После завершения этого шага должно быть реализовано и зафиксировано:

- где лежат seed-данные
- как данные попадают в `localStorage`
- какие ключи используются
- какой сервис отвечает за `localStorage`
- какой сервис отвечает за инициализацию
- что `localStorage` является основным источником истины после первого запуска
- что все изменения сохраняются локально и не теряются после перезагрузки
