## Шаг 5. Каталог проектов и поиск

Нужно реализовать главную страницу со списком проектов.  
Эта страница должна быть простой, понятной и сразу показывать структуру данных приложения.

Маршрут страницы:

`/projects`

Именно с этой страницы пользователь должен начинать работу с приложением.

---

## Назначение страницы

Страница каталога проектов нужна для:

- просмотра всех проектов
- быстрого поиска
- перехода в конкретный проект
- понимания общей структуры задач и времени

Это не должна быть перегруженная dashboard-страница.  
Нужен чистый и понятный список проектов с базовой полезной информацией.

---

## Что должно быть на странице

На странице должны быть следующие элементы:

- заголовок страницы
- строка поиска
- список проектов
- кнопка создания проекта
- пустое состояние, если проектов нет

---

## Поиск

На странице нужен один общий поиск.

### По каким полям должен работать поиск

Поиск должен уметь находить совпадения по:

- названию проекта
- коду проекта
- коду задачи
- названию задачи

### Как должен работать поиск

- поиск должен быть локальным
- поиск должен работать без backend
- поиск должен обновлять список сразу при вводе
- поиск должен быть нечувствителен к регистру
- поиск должен искать по всем доступным проектам и связанным задачам

### Ожидаемое поведение

Если пользователь вводит:

- `PRO1` → находится проект с кодом `PRO1`
- `PRO1-003` → находится проект, внутри которого есть такая задача
- часть названия задачи → также должен находиться соответствующий проект

### Что важно

На странице показываются именно **проекты**, а не задачи.  
То есть поиск по задачам нужен не для вывода задач здесь, а для того, чтобы отфильтровать список проектов, к которым эти задачи относятся.

---

## Карточка проекта

Каждый проект на странице должен отображаться в виде отдельной карточки или строки.

### Что должно быть видно у проекта

- название проекта
- код проекта
- короткое описание, если есть
- цвет проекта
- количество задач
- общее потраченное время по проекту
- кнопка или ссылка “Открыть”

### Общее время проекта

На карточке нужно показывать суммарное количество часов по всем задачам проекта.

Это значение должно вычисляться автоматически на основе worklog-записей.

---

## Визуальное поведение карточки проекта

Карточка должна быть простой, аккуратной и читабельной.

### Требования

- код проекта должен хорошо выделяться визуально
- цвет проекта должен использоваться как акцент
- карточка должна быть кликабельной целиком или иметь явную кнопку перехода
- не перегружать карточку лишними метаданными

---

## Создание проекта

На странице должна быть возможность создать новый проект.

### Минимальные поля проекта

- `name`
- `code`
- `description` — опционально
- `color`

### Требования к созданию проекта

- форма может быть встроенной или открываться в модальном окне
- `code` должен быть уникальным
- после создания новый проект должен сразу появляться в каталоге
- проект должен сохраняться в `localStorage`

### Что не нужно

- не нужна сложная валидация
- не нужны дополнительные настройки проекта
- не нужны архивы, удаление и редактирование на этом этапе

---

## Пустое состояние

Если проектов нет, нужно показать пустое состояние.

### Пустое состояние должно содержать

- понятный текст
- кнопку создания проекта

### Но

Так как в проекте используются seed-данные, пустое состояние в обычном демо-режиме будет встречаться редко.  
Тем не менее его нужно предусмотреть.

---

## Данные для отображения

Для построения каталога страница должна использовать:

- список проектов
- список задач
- список worklog-записей

### Что должно вычисляться

Для каждого проекта нужно вычислять:

- количество задач
- общее время по проекту

Эти вычисления не должны делаться вручную в шаблоне.  
Их нужно вынести в сервис или подготовить заранее в view-model.

---

## Рекомендуемая структура страницы

Страница может состоять из следующих частей:

- `page-header`
- `search-input`
- `project-list`
- `project-card`
- `empty-state`

---

## Поведение при клике

При клике на проект нужно переходить на страницу проекта:

`/projects/:id`

---

## Что не нужно делать на этом шаге

- не выводить список задач прямо в каталоге
- не добавлять аналитику по неделям на эту страницу
- не делать сложные фильтры по статусам
- не делать сортировку по множеству параметров
- не превращать каталог в dashboard

---

## Ожидаемый результат шага

После завершения этого шага должно быть реализовано:

- страница `/projects`
- отображение списка проектов
- локальный поиск по проектам и задачам
- показ количества задач по каждому проекту
- показ общего времени по каждому проекту
- создание нового проекта
- переход на страницу проекта
