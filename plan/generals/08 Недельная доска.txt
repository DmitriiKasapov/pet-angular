## Шаг 8. Недельная доска

Нужно реализовать отдельную страницу недельного планирования и учёта времени, которая будет одной из ключевых частей MVP.

Маршрут страницы:

`/board`

Эта страница должна показывать, что приложение умеет не только хранить проекты и задачи, но и работать с недельным планом, распределением задач по дням и фиксацией затраченного времени.

---

## Назначение страницы

Недельная доска нужна для:

- просмотра рабочей недели
- распределения задач по дням
- добавления worklog-блоков
- перемещения блоков между днями
- изменения порядка блоков внутри дня
- фиксации времени работы по задачам

---

## Общая структура страницы

На странице нужно показать недельную доску с 7 колонками:

- понедельник
- вторник
- среда
- четверг
- пятница
- суббота
- воскресенье

Каждая колонка представляет один день недели.

---

## Верхняя панель страницы

В верхней части страницы нужно разместить элементы управления неделей.

### Что должно быть в верхней панели

- заголовок страницы
- отображение текущей недели
- кнопка перехода к предыдущей неделе
- кнопка перехода к следующей неделе
- кнопка возврата к текущей неделе
- кнопка добавления нового worklog-блока

---

## Переключение недели

Нужно реализовать работу с неделями.

### Требования

- доска всегда показывает одну выбранную неделю
- можно переключаться на предыдущую и следующую неделю
- можно быстро вернуться к текущей неделе
- при переключении недели должны отображаться только записи этой недели

### Важно

Неделя должна строиться как набор из 7 календарных дней, начиная с понедельника.

---

## Колонки дней

Каждая колонка должна отображать один день недели.

### Что должно быть видно в колонке

- название дня
- дата
- список worklog-блоков этого дня
- пустое состояние, если записей нет

### Поведение

- внутри каждой колонки должны отображаться только записи выбранной недели и соответствующего дня
- блоки должны быть отсортированы по времени

---

## Worklog-блок

Каждая запись worklog должна отображаться в виде отдельной карточки на доске.

### Что должно быть видно в блоке

- код задачи
- название задачи
- код проекта или название проекта
- статус задачи
- startHour
- durationHours
- endHour
- comment, если есть

### Вычисление endHour

`endHour` не хранится отдельно в данных.  
Он должен вычисляться автоматически:

`endHour = startHour + durationHours`

### Цветовая индикация

Блок должен быть визуально связан со статусом задачи или проектом.

Допустимо использовать:

- цвет статуса
- цвет проекта
- оба варианта вместе, если это не перегружает интерфейс

---

## Создание worklog-блока

На странице должна быть возможность создать новый worklog-блок.

### Способ создания

Можно использовать:

- модальное окно
- боковую панель
- встроенную форму

Для MVP допустим любой простой и быстрый вариант.

---

## Поля при создании блока

При создании worklog-блока пользователь должен указать:

- задачу из списка
- день недели
- `startHour`
- `durationHours`
- `comment` — опционально

### Выбор задачи

Задача должна выбираться из существующего списка задач.

### Ограничение по задачам

В списке выбора не должны показываться задачи со статусом `done`.

### Важно

Одна и та же задача может быть добавлена на доску несколько раз в течение недели.

---

## Drag and drop

На доске нужно реализовать drag and drop.

### Что должно поддерживаться

- перемещение блока между днями
- изменение порядка блоков внутри одного дня

### Требования

- использовать Angular CDK DragDrop
- после перемещения данные должны сохраняться в `localStorage`
- если блок перенесли в другой день, нужно обновить:
  - `dayIndex`
  - `date`
- если изменился порядок внутри дня, это должно сразу отражаться в интерфейсе

### Упрощение для MVP

Не нужно реализовывать drag-resize по времени.  
Не нужно реализовывать перенос по часовым слотам мышкой.

---

## Сортировка внутри дня

Блоки внутри дня должны отображаться в логичном порядке.

### Базовое правило

Сортировка должна выполняться по `startHour`.

### После drag and drop

Если внутри дня допустимо ручное изменение визуального порядка, итоговый порядок всё равно должен оставаться понятным.  
Для MVP допустимо после каждого изменения заново сортировать блоки по `startHour`.

---

## Пустое состояние дня

Если в колонке нет ни одной записи, нужно показать простое пустое состояние.

Пример:
- “Нет записей”
- или просто минимальный placeholder

---

## Данные, нужные для доски

Для построения доски страница должна использовать:

- worklog-записи
- задачи
- проекты

### Что нужно подготовить заранее

Для каждой записи доски удобно подготовить view-model, в которой уже есть:

- данные записи
- данные задачи
- данные проекта
- вычисленный `endHour`
- цветовой статус

---

## Загрузка страницы

При открытии страницы `/board` нужно:

- определить текущую выбранную неделю
- получить все worklog-записи этой недели
- связать их с задачами
- связать задачи с проектами
- сгруппировать записи по 7 дням недели
- отсортировать записи внутри каждого дня

---

## Что не нужно делать на этом шаге

- не делать месячный календарь
- не делать почасовую сетку как в Google Calendar
- не делать resize блока мышкой
- не делать multi-user planning
- не делать фильтрацию по пользователям
- не добавлять сложные зависимости между задачами

---

## Ожидаемый результат шага

После завершения этого шага должно быть реализовано:

- страница `/board`
- отображение выбранной недели
- 7 колонок Пн–Вс
- показ worklog-блоков по дням
- создание нового worklog-блока
- выбор задачи из списка
- исключение задач со статусом `done` из выбора
- вычисление `endHour`
- drag and drop между днями
- drag and drop внутри дня
- сохранение изменений в `localStorage`
