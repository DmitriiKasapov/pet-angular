## Шаг 9. Логика времени и правила расчётов

Нужно заранее зафиксировать единые правила работы со временем, чтобы недельная доска, страница задачи, страница проекта и аналитика считали часы одинаково и без расхождений.

Все расчёты в приложении должны быть простыми, предсказуемыми и прозрачными.  
Для MVP не нужно делать сложную календарную математику.  
Нужна понятная система учёта часов.

---

## Основные поля времени

Для worklog-записи используются следующие поля:

- `date`
- `dayIndex`
- `startHour`
- `durationHours`

### Вычисляемое поле

`endHour` не хранится в данных.  
Оно вычисляется по формуле:

`endHour = startHour + durationHours`

---

## Единицы измерения

В MVP все часы учитываются в целых числах.

### Правила

- `startHour` — целое число
- `durationHours` — целое число
- `endHour` — целое число
- дробные значения не использовать
- шаг времени — 1 час

### Пример

- `startHour = 9`
- `durationHours = 3`
- `endHour = 12`

---

## Диапазон времени

Нужно использовать фиксированный рабочий диапазон времени.

### Рекомендуемый диапазон

- от `8` до `20`

Это значит:

- минимальный `startHour` = 8
- максимальный допустимый `endHour` = 20

### Следствие

При создании worklog-записи нужно проверять, чтобы:

- `startHour >= 8`
- `durationHours > 0`
- `endHour <= 20`

---

## Правила для создания записи

При создании новой worklog-записи пользователь задаёт:

- день недели
- `startHour`
- `durationHours`
- комментарий — опционально
- задачу из списка

### Что должно вычисляться автоматически

- конкретная календарная `date` внутри выбранной недели
- `dayIndex`
- `endHour`

---

## Логика date и dayIndex

Для каждой записи должны храниться:

- календарная дата
- индекс дня недели

### Правило

`date` — это фактическая календарная дата выбранного дня в рамках текущей недели.  
`dayIndex` — техническое поле для привязки к колонке недели.

### Нумерация дней

- 1 — понедельник
- 2 — вторник
- 3 — среда
- 4 — четверг
- 5 — пятница
- 6 — суббота
- 7 — воскресенье

---

## Перемещение записи между днями

При переносе блока в другую колонку через drag and drop нужно обновлять:

- `dayIndex`
- `date`

### Что не должно меняться автоматически

- `startHour`
- `durationHours`

Для MVP перенос между днями меняет только день, но не время начала и не длительность.

---

## Сортировка записей внутри дня

Все записи внутри одного дня должны сортироваться по `startHour`.

### Правило сортировки

Сначала более ранние записи, потом более поздние.

### Пример

- 09:00–11:00
- 12:00–13:00
- 15:00–17:00

### Важно

Сортировка должна быть единообразной во всех местах, где выводятся записи дня.

---

## Пересечения по времени

Для MVP не нужно запрещать пересечение задач по времени.

### Это означает

Допустима ситуация, когда в одном дне есть:

- задача A: 09–11
- задача B: 10–12

### Почему так

- это упрощает MVP
- не требует сложной проверки конфликтов
- позволяет быстрее завершить проект

### Но

Если будет время, можно добавить простую проверку-предупреждение, но это не обязательное требование текущего этапа.

---

## Подсчёт spent hours по задаче

Потраченное время по задаче считается так:

сумма всех `durationHours` у worklog-записей, где:

`WorklogEntry.taskId === Task.id`

### Пример

Если у задачи 3 записи:

- 2 часа
- 1 час
- 4 часа

Итог:

`spentHours = 7`

---

## Подсчёт времени по проекту

Потраченное время по проекту считается так:

1. найти все задачи проекта
2. найти все worklog-записи этих задач
3. сложить `durationHours`

### Формула по смыслу

`projectSpentHours = sum(durationHours for all entries of all project tasks)`

---

## Подсчёт времени по неделе

Общее время за выбранную неделю считается как:

сумма всех `durationHours` у worklog-записей, которые принадлежат выбранной неделе

### Важно

Неделя должна определяться не по номеру недели, а по конкретному диапазону дат:

- начало недели — понедельник
- конец недели — воскресенье

---

## Подсчёт времени по дням недели

Для аналитики по неделе нужно уметь считать часы по каждому дню:

- понедельник
- вторник
- среда
- четверг
- пятница
- суббота
- воскресенье

### Правило

Для каждого дня суммируются `durationHours` всех записей этого дня в рамках выбранной недели.

---

## Подсчёт времени по задачам за неделю

Для недельной аналитики нужно считать:

- сколько часов ушло на каждую задачу в рамках выбранной недели

### Правило

Суммируются только записи выбранной недели.

---

## Подсчёт времени по проектам за неделю

Для недельной аналитики нужно считать:

- сколько часов ушло на каждый проект в рамках выбранной недели

### Правило

1. взять записи выбранной недели
2. определить задачи этих записей
3. связать задачи с проектами
4. просуммировать `durationHours` по каждому проекту

---

## Что считается вычисляемым, а не хранимым

Следующие значения не нужно хранить отдельными полями:

- `endHour`
- `spentHours` по задаче
- `spentHours` по проекту
- часы за неделю
- часы по дням недели
- часы по задачам за неделю
- часы по проектам за неделю

Все эти значения должны вычисляться через сервисы или утилиты.

---

## Где должна жить логика расчётов

Логику расчётов нельзя размазывать по компонентам.

### Требование

Основные вычисления должны быть вынесены в отдельный сервис или набор утилит.

### Допустимые варианты

- `analytics.service.ts`
- `worklog.utils.ts`
- или комбинация сервиса и утилит

### Запрещено

- считать часы прямо в шаблонах
- дублировать одну и ту же формулу в нескольких страницах
- вычислять `endHour` разными способами в разных местах

---

## Единые правила отображения времени

Для MVP достаточно простого текстового формата.

### Примеры отображения

- `09:00 – 12:00`
- `3 ч`
- `Всего: 14 ч`

### Правило

Во всём приложении использовать единый формат отображения времени.

---

## Поведение при изменении данных

После любой операции с worklog-записью должны пересчитываться связанные данные:

- spent hours по задаче
- spent hours по проекту
- часы недели
- часы по дням
- часы по задачам и проектам в аналитике

Пересчёт может быть сделан:
- реактивно в сервисе
- либо повторным вычислением при загрузке view-model

Для MVP допустим любой простой и надёжный вариант.

---

## Что не нужно делать на этом шаге

- не использовать минуты
- не использовать дробные часы
- не делать таймер реального времени
- не делать авто-трекинг
- не делать учёт переработок
- не делать учёт по нескольким пользователям
- не учитывать праздники, отпуска и особые календари

---

## Ожидаемый результат шага

После завершения этого шага должны быть зафиксированы:

- единицы измерения времени
- диапазон рабочих часов
- правило вычисления `endHour`
- правила сортировки внутри дня
- правила подсчёта часов по задаче
- правила подсчёта часов по проекту
- правила подсчёта часов по неделе
- единый подход, где живёт логика расчётов
