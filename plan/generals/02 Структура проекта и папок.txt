## Шаг 2. Структура проекта и папок

Нужно заранее зафиксировать понятную и современную структуру Angular-проекта, чтобы код выглядел аккуратно, масштабируемо и профессионально.  
Структура должна быть простой, без лишней сложности, но не хаотичной.

### Общий подход

Использовать структуру по смысловым зонам приложения:

- pages — страницы
- features — прикладные сущности и бизнес-логика
- shared — переиспользуемые компоненты, утилиты и общие типы
- core — базовые сервисы приложения
- data — seed-данные и константы

### Назначение основных папок

#### core
Здесь хранятся базовые сервисы, которые используются во всём приложении.

Примеры:
- работа с `localStorage`
- инициализация seed-данных
- агрегирующая аналитика

#### data
Здесь лежат стартовые демо-данные.

Примеры:
- проекты
- задачи
- worklog-записи
- комментарии

#### models
Здесь описываются все основные типы и интерфейсы данных приложения.

#### features
Здесь хранится логика по отдельным предметным зонам:
- проекты
- задачи
- worklog

Если для конкретной зоны появятся утилиты или отдельные сервисы, они должны лежать внутри соответствующей feature-папки.

#### pages
Здесь лежат именно страницы маршрутов приложения.

Каждая страница должна быть standalone component и иметь свою отдельную папку.

#### shared
Здесь лежит всё переиспользуемое:
- простые UI-компоненты
- общие строительные блоки интерфейса
- утилиты
- константы

### Страницы приложения

Нужно предусмотреть следующие страницы:

- `/projects` — каталог проектов
- `/projects/:id` — страница проекта
- `/tasks/:id` — страница задачи
- `/board` — недельная доска
- `/analytics` — недельная аналитика

### Требования к страницам

- каждая страница должна быть standalone component
- каждая страница должна иметь отдельную папку
- внутри страницы допустимо держать только код, относящийся к этой странице
- общие элементы нельзя дублировать между страницами, их нужно выносить в `shared`

### Правила организации кода

- не складывать всё в одну папку
- не смешивать страницы, модели и сервисы в одном месте
- не создавать слишком глубокую вложенность
- не усложнять архитектуру ради архитектуры
- не использовать старый модульный стиль с множеством NgModule
- не использовать “god service”, который делает всё сразу

### Базовые принципы

- страницы отвечают за отображение и сборку экрана
- сервисы отвечают за данные и бизнес-логику
- модели отвечают за структуру сущностей
- shared-компоненты отвечают за переиспользуемый UI
- storage-логика не должна размазываться по страницам

### Результат шага

После завершения этого шага должна быть зафиксирована структура проекта:

- понятное дерево папок
- список основных страниц
- разделение на core, data, models, features, pages и shared
- единый принцип, где должен лежать каждый тип кода
