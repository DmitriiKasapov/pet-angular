# Правила для Claude

## Рабочий процесс

**В начале КАЖДОЙ сессии обязательно прочитать:**

1. `/plan/work.md` — текущие задачи
2. `/plan/ended.md` — что уже сделано
3. `/plan/generals/` — концепция проекта, стек, общие правила

Работа ведётся согласно структуре папки `/plan`:

- **Концепция и стек** — `/plan/generals/`
- **Текущие задачи** — `/plan/work.md`
- **Отчёт о выполнении** — `/plan/ended.md`

### Обязательное правило

После выполнения **каждого шага** или завершения значимой подзадачи нужно сделать краткую запись в `/plan/ended.md`.

Формат записи:

- дата
- название шага / задачи
- что сделано
- статус

Пример:

- `2026-04-01 — Шаг 5. Каталог проектов — реализована страница /projects, поиск и карточки проектов — done`

---

## Общие принципы разработки

1. Проект разрабатывается **по актуальным современным требованиям**.
2. Использовать **последние стабильные и уместные для проекта подходы**, а не устаревшие практики.
3. Если решение **спорное, неоднозначное или архитектурно влияет на проект**, не принимать его молча:
   - кратко описать проблему
   - предложить 1–2 разумных варианта
   - зафиксировать это для обсуждения
4. Не усложнять проект без необходимости.
5. MVP должен оставаться **реалистичным, компактным и демонстрационным**, а не превращаться в большой production-сервис.

---

## Язык общения и кодовые правила

1. Всё общение с разработчиком — **только на русском языке**.
2. Все комментарии в коде — **только на английском языке**.
3. Все названия сущностей, переменных, интерфейсов, типов, функций, файлов и компонентов — **на английском языке**.
4. Технические описания в документации проекта можно писать на русском, если это внутренняя документация для разработки.
5. **Весь пользовательский интерфейс приложения — только на английском языке.** Это включает: текстовые метки, заголовки, подписи, placeholder-ы, сообщения об ошибках, пустые состояния, кнопки, табы и seed-данные (названия проектов, задач, комментарии).

## Технический стек проекта

Использовать современный стек проекта:

- Angular
- TypeScript
- Tailwind CSS
- Angular Router
- Reactive Forms
- Angular CDK DragDrop
- Signals — там, где это уместно для локального состояния
- localStorage + seed data для хранения данных в MVP

### Архитектурные требования

- использовать **standalone components**
- не строить архитектуру вокруг устаревшего NgModule-подхода
- не использовать тяжёлые решения без необходимости
- не добавлять backend или реальную БД в рамках MVP
- не добавлять лишние библиотеки, если задача решается средствами Angular и текущего стека

---

## Хранение данных

Для MVP используется только локальное хранение данных:

- начальные данные лежат в seed-файлах
- при первом запуске seed копируется в `localStorage`
- далее приложение работает только с `localStorage`

### Правила

- не обращаться к `localStorage` напрямую из компонентов
- вся логика хранения должна быть вынесена в сервисы
- seed-инициализация должна быть отдельной и прозрачной
- после любого изменения данные должны сохраняться локально

## Структура компонентов и папок

### Структура `src/app/`

```
src/app/
  components/        ← переиспользуемые компоненты (shared)
    blocks/          ← крупные секции, используемые на нескольких страницах
    modules/         ← модули с логикой, используемые в нескольких местах
    elements/        ← атомарные UI-элементы (button, badge, input, empty-state)
  pages/             ← страницы маршрутов
    projects-page/
      blocks/        ← секции этой страницы
      modules/       ← модули этой страницы
    project-detail-page/
      blocks/
      modules/
    task-detail-page/
      blocks/
      modules/
    weekly-board-page/
      blocks/
      modules/
    analytics-page/
      blocks/
      modules/
  core/
    services/        ← все сервисы: storage, seed, analytics, projects, tasks, worklog
  data/
    seed/            ← seed-файлы с демо-данными
  models/            ← интерфейсы сущностей
```

### Категории компонентов

**`blocks/`** — логические блоки-секции страницы. Используют тег `<section>`. Крупные, самодостаточные.  
Примеры: `ProjectList`, `TaskBoard`, `AnalyticsSummary`

**`modules/`** — самостоятельные модули с логикой. Формы, карточки, списки.  
Примеры: `ProjectCard`, `TaskForm`, `WorklogForm`

**`elements/`** — атомарные элементы без бизнес-логики.  
Примеры: `Button`, `Badge`, `EmptyState`, `Spinner`

### Правила именования корневого класса

Корневой CSS-класс строится по пути от `src/app/`, через `__`, в kebab-case.

- `pages/projects-page/blocks/project-list/` → `projects-page__blocks__project-list`
- `pages/project-detail-page/modules/task-form/` → `project-detail-page__modules__task-form`
- `components/elements/badge/` → `components__elements__badge`

### Правила размещения

- Компонент нужен только одной странице → в `pages/<page>/blocks/` или `pages/<page>/modules/`
- Компонент нужен нескольким страницам → в `components/blocks/`, `components/modules/` или `components/elements/`
- Нельзя класть компоненты в корень страницы без подпапки категории
- Нельзя дублировать компонент между страницами — нужно переносить в `components/`

## Стили и Tailwind CSS

### Структура CSS

Стили разбиты по папкам внутри `src/styles/`:

```
src/styles/
  base/
    theme.css        ← все дизайн-токены в @theme {}
    fonts.css        ← подключение шрифтов через @font-face
    base.css         ← сброс и базовые стили HTML-элементов
    typography.css   ← стили заголовков и текста
  utilities/
    layout.css       ← утилиты для layout (если нужны)
  components/
    buttons.css      ← .btn, .btn-primary, .btn-secondary и т.д.
    inputs.css       ← .input, состояния, ошибки
    badges.css       ← .badge и варианты статусов
    cards.css        ← .card базовый стиль
```

Главный файл `src/styles.scss` импортирует всё в нужном порядке:

```scss
@import "tailwindcss";

@import "./styles/base/theme.css";
@import "./styles/base/fonts.css";
@import "./styles/base/base.css";
@import "./styles/base/typography.css";

@import "./styles/components/buttons.css";
@import "./styles/components/inputs.css";
@import "./styles/components/badges.css";
@import "./styles/components/cards.css";
```

### Дизайн-токены (@theme)

Все токены объявляются в `@theme {}` — это Tailwind v4-подход. Никаких magic-значений в шаблонах.

```css
@theme {
  /* Colors — semantic */
  --color-primary: #3b5bdb;
  --color-primary-hover: #364fc7;
  --color-surface: #ffffff;
  --color-bg: #f8f9fa;
  --color-border: #e9ecef;
  --color-text: #212529;
  --color-text-muted: #868e96;

  /* Status colors */
  --color-status-planning: #74c0fc;
  --color-status-progress: #ffa94d;
  --color-status-done: #69db7c;

  /* Typography */
  --font-sans: Arial, system-ui, sans-serif;

  /* Shadows */
  --shadow-card: 0 1px 4px rgba(0,0,0,0.08);
}
```

### Компонентные классы

Переиспользуемые классы пишутся через `@layer components` в отдельных файлах:

```css
@layer components {
  .btn { @apply inline-flex items-center gap-2 px-4 py-2 rounded-lg font-medium transition-colors; }
  .btn-primary { @apply bg-primary text-white hover:bg-primary-hover; }
  .btn-secondary { @apply bg-transparent border border-border text-text hover:bg-bg; }
  .btn-ghost { @apply bg-transparent text-text-muted hover:text-text hover:bg-bg; }
}
```

### Принципы

- **Никаких inline-стилей** — только Tailwind-классы или компонентные классы
- **Никаких магических значений** — только токены из `@theme`
- **Варианты через match/object** в Angular-компонентах, не условные классы в шаблоне
- **Кастомный grid не используется** — приложение, не маркетинговый сайт; `flex` и стандартный `grid` достаточно

### Дизайн-принципы

- Стиль: **минималистичный, деловой, user-friendly**
- Шрифт: **Arial / system-ui** — без кастомных шрифтов
- Без декоративных эффектов и анимаций без необходимости
- Единый визуальный стиль во всём приложении: кнопки, инпуты, карточки, статусы, пустые состояния

## Кодовые правила

1. Писать код чисто и последовательно.
2. Не дублировать логику в нескольких местах.
3. Не размещать бизнес-логику в шаблонах.
4. Не размещать storage-логику в компонентах страниц.
5. Вычисления, агрегации и аналитику выносить в сервисы или утилиты.
6. Не использовать inline styles без крайней необходимости.
7. Для стилизации использовать Tailwind CSS.
8. Не создавать хаотичную структуру файлов.
9. Каждый новый файл должен появляться в логичном месте структуры проекта.
10. Не писать “временные решения”, если они ломают читаемость архитектуры.

---

## UI и стили

1. Интерфейс должен быть аккуратным, современным и чистым.
2. Не перегружать UI декоративными эффектами.
3. Использовать единый визуальный стиль для:
   - кнопок
   - инпутов
   - карточек
   - табов
   - статусов
   - пустых состояний
4. Статусы задач должны иметь единый цветовой стиль во всём приложении.
5. Пустые состояния и состояния “не найдено” должны быть предусмотрены.
6. Не использовать случайные визуальные решения от экрана к экрану.

---

## Подход к реализации

1. Работать **пошагово**, в рамках текущей спеки.
2. Не забегать далеко вперёд без необходимости.
3. Если текущий шаг не завершён — не расползаться на другие крупные части проекта.
4. При реализации опираться на:
   - текущую спеку
   - уже завершённые шаги
   - ограничения MVP
5. Если в процессе становится видно, что решение перегружает MVP, это нужно отметить и предложить упрощение.

---

## Работа со спорными решениями

Если возникает спорный момент, например:

- архитектура сервиса
- хранение вычисляемых данных
- форма drag and drop поведения
- организация состояния
- UI-паттерн, который можно сделать по-разному

то нельзя просто выбирать случайный путь.

Нужно:

1. коротко обозначить спорный момент
2. предложить разумное решение
3. при необходимости отметить альтернативу
4. зафиксировать это прозрачно

---

## Документация выполнения

После завершения каждого шага обязательно:

1. обновить `/plan/ended.md`
2. кратко зафиксировать:
   - что было сделано
   - что было изменено
   - завершён ли шаг полностью
3. если шаг завершён не полностью — явно указать, что осталось

---

## Запрещено

1. Молча принимать архитектурно спорные решения.
2. Использовать устаревшие практики, если есть современный и уместный вариант.
3. Раздувать MVP сверх согласованного объёма.
4. Переносить бизнес-логику в шаблоны.
5. Делать прямую работу с `localStorage` из компонентов.
6. Добавлять библиотеки “на всякий случай”.
7. Смешивать русский язык и английский в кодовой структуре.
8. Оставлять выполненные шаги без фиксации в `/plan/ended.md`.

---

## Результат работы Claude

Каждая сессия должна приводить к понятному результату:

- выполнен конкретный шаг
- изменения согласованы с текущим планом
- прогресс зафиксирован в `/plan/ended.md`
- проект остаётся чистым, современным и пригодным для показа
